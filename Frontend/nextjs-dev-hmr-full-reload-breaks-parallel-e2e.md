# next dev 위에서 병렬 e2e를 돌리면 HMR 풀 리로드가 테스트를 죽인다

Next.js dev 서버를 webServer로 쓰는 Playwright 병렬 스위트에서, **단독 실행은 100% 통과하는 테스트가 전체 실행에서만 무작위로 실패**하는 문제를 추적했다. 드롭다운/모달의 확정 클릭이 아무 흔적 없이 유실되는 증상 — 낙관적 UI 반영도, 토스트도, 에러도 없었다.

## 범인 추적

기각한 가설들:

1. **서버 응답 지연** — 단언 타임아웃을 15s → 30s로 늘려도 재현. 느린 게 아니라 아예 안 일어난 것
2. **테스트 간 데이터 레이스** — 일부는 실제 레이스였지만(공유 시드 행), 어떤 테스트는 공유 데이터가 전혀 없는데도 실패
3. **일시적 플레이크** — retry를 줬는데 3연속 실패하는 런이 나옴. 일시적이지 않음

`--trace on`으로 실패 시점의 브라우저 콘솔 타임라인을 뜯자 답이 나왔다:

```
[HMR] connected            ← 페이지 로드
[Fast Refresh] rebuilding  ← 아무 파일도 안 바꿨는데?
[Fast Refresh] done in 1217ms
[HMR] connected            ← ★ 재연결 = 페이지가 풀 리로드됨
```

**`[HMR] connected`가 두 번 찍히면 그 사이에 풀 리로드가 있었다는 뜻이다.**

## 원인

next dev는 라우트를 **최초 요청 시점에 지연 컴파일**한다. 병렬 워커들이 서로 다른 라우트를 처음 밟을 때마다 리빌드가 일어나고, 이 리빌드는 HMR 웹소켓으로 **열려 있는 모든 페이지에 방송**된다. 공유 청크가 바뀌면 Fast Refresh가 핫 패치 대신 **풀 리로드**를 수행한다.

그 순간 다른 워커의 페이지에서 상호작용 중이던 오버레이(Radix 드롭다운·모달)는 통째로 언마운트되고, 진행 중이던 클릭은 어떤 핸들러에도 도달하지 못한다. 파일 변경이 없는 e2e 런에서 리빌드 유발원은 콜드 라우트 컴파일뿐이라, **어떤 워커가 어떤 라우트를 먼저 밟느냐에 따라 피해자가 매번 달라진다** — "무작위 스펙이 걸리는" 이유.

## 해법: 라우트 워밍업

테스트 스위트가 밟는 모든 라우트를 **병렬 스펙 시작 전에 직렬로 한 번씩 방문**하는 setup 프로젝트를 둔다:

```ts
// e2e/warmup.setup.ts — 컴파일 유발이 목적이라 best-effort(네비 에러 무시·유한 타임아웃)
const ROUTES = ['/', '/a', '/b', ...];
test('warm up dev-server routes', async ({ page }) => {
  for (const route of ROUTES) {
    await page.goto(route, { waitUntil: 'load', timeout: 45_000 }).catch(() => {});
  }
});
```

```ts
// playwright.config.ts — 본 스펙 프로젝트가 warmup에 의존
{ name: 'warmup', testMatch: /warmup\.setup\.ts/, dependencies: ['setup'] },
{ name: 'chromium', dependencies: ['setup', 'warmup'], ... },
```

콜드 컴파일이 전부 소진된 뒤에는 (파일 변경이 없으므로) 리빌드도 리로드도 없다. 완화용으로 넣었던 retry를 걷어내고도 스위트가 반복 green — retry 없이 통과시키는 것이 근본 수정의 증명이 된다.

## 곁가지 함정 두 개

- **`fullyParallel: true`는 같은 파일 안의 테스트도 병렬로 돌린다.** 파일 단위 직렬이라고 가정하고 시드 행·마커 값을 파일 내에서 공유하면 그것대로 레이스가 난다 — 테스트별로 격리할 것
- **오래 살아 있는 dev 서버는 병렬 스위트를 반복해서 받으면 메모리 압박으로 전방위 타임아웃을 일으킨다.** 런타임이 점점 늘어나면(3.1m → 4.4m) 서버 퍼짐 신호 — 재시작이 답

프로덕션 빌드(`next build && start`)로 돌리면 HMR 자체가 없어 이 문제가 원천 차단되지만, 로컬 반복 개발 흐름과의 트레이드오프가 있어 워밍업이 더 싼 해법이었다.
