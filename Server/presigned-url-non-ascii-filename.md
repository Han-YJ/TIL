# Presigned URL 업로드가 파일명에 따라 간헐 실패한다면 — 비ASCII 인코딩

S3류 스토리지의 presigned URL은 **서명 생성에 들어간 값(오브젝트 키·헤더)과 실제 요청이 정확히 일치**해야 한다. 하나라도 어긋나면 서명 불일치로 400/403이 난다.

## 증상 패턴

- 발급 API(백엔드)는 200으로 signedUrl을 잘 내려줌
- 클라이언트가 그 URL로 PUT하면 **어떤 파일은 성공, 어떤 파일은 400** — 간헐 실패처럼 보임
- 서버 로그에는 아무것도 안 남음 (PUT은 브라우저 → 스토리지 직행이라 백엔드를 안 거침)

## 원인 격리 과정

1. 실패 지점 이분: 발급 단계 vs PUT 단계 → 발급 200이므로 PUT
2. 파일 타입 가설: 같은 확장자(pdf)인데도 성공/실패 갈림 → 기각
3. 사용자 액션 타이밍 가설: 무관한 조작 중에도 재현/비재현 → 기각
4. 성공·실패 파일의 차이를 **파일명**으로 좁힘 → **한글 파일명만 실패, 영문은 성공**

## 원인

백엔드가 오브젝트 키를 `{uuid}_{원본파일명}` 식으로 조합하면서 비ASCII 문자를 인코딩하지 않고 서명 → 실제 PUT 시점의 URL 인코딩과 어긋나 서명 불일치.

## 교훈

- 오브젝트 키에 사용자 입력(파일명)을 넣지 말 것 — **키는 uuid로 정규화하고 원본 파일명은 메타데이터로만 보존**이 안전
- "간헐 실패"는 대개 간헐이 아니다 — 성공/실패 표본의 차이 나는 축(파일명·크기·타입·타이밍)을 하나씩 소거하면 결정적 조건이 나온다
- Content-Type도 같은 원리 — 발급 때 서명에 들어갔다면 PUT 헤더와 반드시 일치시켜야 한다
