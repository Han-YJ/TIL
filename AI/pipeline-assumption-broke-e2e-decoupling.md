# 파이프라인 끝에 붙인 e2e가, 전제가 깨지자 부채가 됐다

AI 스킬로 화면 개발 파이프라인을 짜면서 e2e를 이렇게 배치했다.

```
계획 → 스캐폴딩 → 구현 → 디자인 정합 검증 → 코드 리뷰 → e2e 테스트
```

맨 끝에 e2e를 둔 건 자연스러운 선택이었다. **"화면이 완성됐다 = API까지 붙어 동작한다"** 는 암묵적 전제가 있었기 때문이다.

## 필요성 발견 — 전제가 깨졌다

실제 개발은 그렇게 흐르지 않았다. **UI를 먼저 만들고, API 연동은 별도 티켓으로 뒤에 붙는** 경우가 많았다. 백엔드가 아직 없거나, 다른 팀 일정과 어긋나거나, 화면을 먼저 보여줘야 하는 상황이 계속 생겼다.

그러면 파이프라인 끝의 e2e는 **목(mock) 데이터 위에서** 실행된다. 그런데도 통과한다. 통과하니까 게이트를 넘어가고, 그대로 머지된다.

몇 달 뒤 API가 붙는 순간, 그 e2e들이 **한꺼번에 전부 깨졌다.**

## 문제 — green이 오히려 부채를 쌓고 있었다

깨진 테스트를 열어보니 원인이 명확했다.

```ts
// 목 데이터: 회사명은 '대한특수강', 담당자는 '홍길동'
await expect(page.getByText('대한특수강')).toBeVisible();
```

목 픽스처의 **값 자체를 기대값으로 하드코딩**해뒀다. 실제 API가 붙으면 그 데이터는 존재하지 않는다.

여기서 두 가지를 알게 됐다.

1. **진짜 죄는 "목 단계에 e2e를 쓴 것"이 아니었다.** 같은 시기에 쓴 로그인 가드·폼 검증·라우팅 e2e는 API가 바뀌어도 멀쩡했다. 데이터를 보지 않기 때문이다. 죄는 **목의 값을 검증한 것**이었다.
2. **개인의 실수가 아니라 규칙의 구멍이었다.** 파이프라인에는 "gateway를 목으로 두고 구현" 옵션이 있었고, 게이트는 "e2e green 필수"를 강제했다. 두 규칙이 만나면 *목 위에서 통과한 e2e가 정식 게이트를 만족시킨다.* 시스템이 그렇게 하도록 유도하고 있었다.

## 해결 — e2e를 파이프라인에서 떼어내되, 미룬 걸 기록한다

### 1. 실행 조건을 명시적 술어로 정의

```
E2E_READY(도메인) = 실제 API 연동됨  &&  그 환경에 데이터를 만들 경로가 있음
```

판정 근거는 **코드**다 (주석은 썩는다 — "아직 목입니다"라고 적힌 파일이 이미 실 API를 호출하고 있었다).

### 2. 미루는 단위를 "도메인"이 아니라 "인수조건"으로 자른다

| 인수조건 유형 | 예 | E2E_READY=false일 때 |
|---|---|---|
| **데이터 비의존** | 접근 가드, 라우팅, 폼 검증, 빈 상태, 모달 열림 | **지금 작성** — API가 붙어도 안 깨진다 |
| **데이터 의존** | 목록 내용, 검색·정렬 결과, 상세 값, 건수 | **미룬다** (기록 필수) |

이게 핵심이다. "API 없으니 e2e는 나중에"가 아니라, **지금 쓸 수 있는 절반은 지금 쓴다.** 재작업 비용이 0이면서 회귀는 조기에 잡는다.

### 3. 하드룰 하나

> **목 픽스처의 값을 테스트에서 assert하지 않는다.** 데이터가 필요하면 시드가 만든 값을 참조한다.

### 4. 미룬 것을 "조용한 스킵"이 되지 않게

- 목 데이터로 구현 중이면 e2e 게이트 결과는 자동으로 **`deferred`** — green으로 통과시키지 않는다
- `deferred`는 **원장(커밋되는 파일) + 티켓 키가 있어야만** 유효하다. 없으면 게이트가 거부한다
- 한 번 `green`이던 영역이 `deferred`로 되돌아가면 **회귀로 판정**한다

이렇게 하면 "나중에 하자"가 사라지지 않고 **보드와 저장소에 남는다.**

### 5. 미룬 걸 소진하는 주체를 만든다

API 연동 티켓이 끝나면, 그 도메인의 테스트 티켓이 밀린 인수조건을 소진한다. 이때 데이터는 **제품이 실제로 쓰는 생성 API를 태워서** 만든다(목 주입이 아니라). 그러면 시드 자체가 계약 검증이 된다.

## 일반화

**파이프라인의 단계 순서는 언제나 어떤 전제 위에 서 있다.** 내 경우 "완성 = API까지 동작"이었고, 그 전제가 깨진 뒤에도 순서만 그대로 남아 몇 달간 부채를 쌓았다.

전제가 깨졌을 때 할 일은 단계를 지우는 것도, 억지로 실행하는 것도 아니다.

1. 전제를 **명시적 술어**로 꺼내고 (`E2E_READY = …`)
2. 그 술어가 거짓일 때 **무엇이 여전히 가능한지** 쪼개고 (데이터 비의존 vs 의존)
3. 못 하는 것은 **기록을 강제**해서 (원장 + 티켓 + 게이트 거부) 조용히 사라지지 않게 한다.

"green인데 부채가 쌓이는" 상태가 가장 위험하다. 실패는 눈에 띄지만, **의미 없는 성공은 눈에 띄지 않는다.**
